Git 极简教程 — 给完全不懂编程的你

写给 Zeno：你是一个奥数老师，刚搞清楚"分支"和"PR"是什么意思。 这篇用大白话讲 Git，不讲术语，只讲你能立刻用的东西。
📁 Git 到底解决什么问题
你的文件夹命名法（懂的都懂）

第一次写完一个文件，叫 奥数讲义.docx。
改了一点，怕覆盖旧的，复制一份叫 奥数讲义-最终版.docx。
又改了一点：奥数讲义-最终版2.docx。
再改：打死不改版.docx。
再改：打死不改版-真的不改了.docx。

两周后你自己都分不清哪个是哪个。

Git 就是一个帮你自动管理版本的工具。它给你三样东西：

⏳ 时光机
随时回到任何一个历史版本
🌌 平行宇宙
同时尝试不同做法，互不影响
👥 多人协作
几个人同时改同一份文件，最后能安全合在一起
✅ 不用再想"这个版本别覆盖了"；不用再手动备份文件夹； 想尝试大胆改动？开个分支随便试，坏了就删掉； 电脑被偷了？代码在云端，换个电脑继续干。
🧠 核心概念（全用比喻）
📦 仓库（repository）—— 项目的"历史档案室"

Git 会给你的项目建一个隐藏的档案室，里面存着你的 每一个历史版本。常缩写成 repo。

📸 提交（commit）—— 拍一张"快照"

你觉得"这个状态不错，先存着"，就做一个提交。每次提交 等于给整个项目拍一张照片，附上你写的一句话："我改了啥"。 以后随时可以回到这张照片的时刻。

🌌 分支（branch）—— 平行宇宙

这个比喻你肯定喜欢，毕竟是你搞明白的第一个术语。

你在主线干活，突然想试一个大胆的想法，就开一个新的分支出来。 新分支一开始跟主线一模一样，你在里面随便改，主线那边纹丝不动。 觉得好，合并回去。觉得不好，直接删掉。

🔗 合并（merge）—— 把平行宇宙合并回来

你开了分支试新想法，试成功了，想让它们正式加入主线。 Git 会智能地把你分支上的改动"粘"回主线。

📋 PR / Pull Request —— "你先看看我改的"

做完一版修改后，请求别人审核一下再合并。 自己做项目时，PR 更像一个"正式发布的仪式"——你先在分支上做改动， 然后发起一个 PR，自己看看改了啥，没问题就合并。

☁️ 远程仓库（remote）—— 代码放云端

你电脑（本地）有一个仓库，还可以关联云端仓库（GitHub）。 电脑丢了不怕，换个电脑克隆下来就行了。

📋 暂存区（staging area）—— 先挑要存档的改动

你改了好几个文件，但可能不想一次全部存档，只想先存一部分。 先 暂存（add）要存的文件，人站齐了，再拍照片 （commit）。

💪 日常使用场景
场景1：第一次把项目放到 Git

背景：你有一个文件夹叫 my-project， 想让 Git 开始管理它。

cd my-project          # 进入项目文件夹
git init               # 初始化仓库
git add .              # 把当前所有文件加入暂存区
git commit -m "第一次提交"  # 拍第一张照片

结果：你的项目现在有完整的 Git 仓库了。

场景2：每天做完工作，保存一版
git status             # 看看改了哪些文件
git add .              # 暂存所有改动
git commit -m "完成了加法功能"  # 提交，写说明

结果：明天改坏了，随时可以回来。

场景3：改了一半，先换个任务？
git stash              # 把半成品先存起来
# 去做别的…
git stash pop          # 恢复回来继续

结果：改到一半的东西完好无损地保存着。

场景4：改错了想反悔
# 更安全的方法：先存起来
git stash

结果：文件回到上次提交时的样子。

场景5：想尝试新想法，又怕搞坏
git checkout -b 试试新方案  # 创建+切换新分支
# 在新分支里随便改…
git add . && git commit -m "在新分支上改的内容"
git checkout main          # 切回主线看看

结果：两个平行世界，互不干扰。觉得好就合并： git merge 试试新方案

场景6：多台电脑同步

第一次推：

git remote add origin https://github.com/你的用户名/项目名.git
git push -u origin main

另一台电脑首次拿代码：

git clone https://github.com/你的用户名/项目名.git
# 改完后
git add . && git commit -m "在家改了一版" && git push
🤖 实战：跟 AI Agent 配合
核心原则：AI 干活前，先建分支
git checkout -b ai-实验功能

让 AI 在这个分支上干活。AI 改啥都不影响你原有的 main 分支。 改完你觉得好，再合并。不好，直接删分支。

🔑 记住一个原则：main（或 master）分支是"干净的、确定没问题的版本"。 所有实验、所有 AI 干的活，都在别的分支上做。这样 AI 就算把你的代码搞烂了， 你切换到 main 分支，世界立刻恢复原样。
❓ 常见问题
什么是 .gitignore？

有些文件不应该被 Git 存档——密码文件、 编译生成的文件、系统隐藏文件。在项目根目录建一个 .gitignore，把不需要跟踪的文件写在里面：

# 忽略密码文件
config/keys.json
# 忽略系统文件
.DS_Store
# 忽略编译生成的文件
node_modules/
冲突是啥？遇到怎么办？

你改了一行，别人也改了同一行，Git 不知道该听谁的。

<<<<<<< HEAD
这是我写的版本
=======
这是别人写的版本
>>>>>>> 别人的分支

解决方法：打开文件，决定保留谁的版本， 删掉所有 <<<< >>>> === 标记， 保存，然后 git add . && git commit。

📋 完整流程：从 0 到 1
# 1. 创建仓库
cd ~/我的项目
git init

# 2. 写第一行代码，保存

# 3. 第一次提交
git add .
git commit -m "项目初始化"

# 4. 开新分支
git checkout -b 添加用户登录功能

# 5. 写代码、改完提交
git add .
git commit -m "完成用户登录"

# 6. 切回主线，合并
git checkout main
git merge 添加用户登录功能

# 7. 推到云端
git remote add origin https://github.com/你的用户名/你的项目.git
git push -u origin main
