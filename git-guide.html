Git 极简教程

写给 Zeno：刚搞清楚"分支"和"PR"是什么意思。
这篇用大白话讲 Git，不讲术语，只讲你能立刻用的东西。
📁 Git 到底解决什么问题
你的文件夹命名法

第一次写完一个文件，叫 奥数讲义.docx。
改了一点，怕覆盖旧的，复制一份叫 奥数讲义-最终版.docx。
又改：奥数讲义-最终版2.docx。
再改：打死不改版.docx。
再改：打死不改版-真的不改了.docx。

两周后你自己都分不清哪个是哪个。

Git 就是一个帮你自动管理版本的工具。

⏳
时光机
随时回到任何一个历史版本
🌌
平行宇宙
同时尝试不同做法，互不影响
👥
多人协作
多人改同一文件，最后能安全合并
✅ 不用再纠结"这个版本别覆盖了"；不用手动备份文件夹； 想试试大胆改动？开个分支随便试，坏了就删掉； 电脑被偷？代码在云端，换个电脑继续干。
🧠 核心概念（全用比喻）
📦
仓库（repository）
项目的历史档案室，存着每一个版本。 常缩写成 repo。
📸
提交（commit）
拍一张"快照"。每次 commit 给整个项目拍一张照片， 附上你写的一句话。以后随时可以回到这个时刻。
🌌
分支（branch）
平行宇宙（你最懂的比喻）。在主线干活时开个分支， 在里面随便改，主线纹丝不动。好就合并，不好就删。
🔗
合并（merge）
把分支上的改动"粘"回主线。 Git 自动按变动拼起来。
📋
PR (Pull Request)
你改完一版，请求别人审核一下再合并。 自己做项目时，更像"正式发布的仪式"。
☁️
远程仓库（remote）
代码放云端（GitHub）。电脑丢了不怕， 换个电脑克隆下来继续干。
📋
暂存区（staging area）
先挑要存档的改动，再正式提交。 跟拍合影一样：先叫人站好（add），再拍（commit）。
💪 日常场景
🏗️ 第一次把项目放到 Git
进入项目文件夹→告诉 Git 开始管理→加入所有文件→拍照存档
cd my-project
git init
git add .
git commit -m "第一次提交"

✅ 项目有了完整 Git 仓库，随时可以回到这个时刻

📝 每天做完工作保存一版
git status     # 看看哪些文件改动了
git add .      # 暂存所有改动
git commit -m "完成了加法功能"  # 拍照+写说明

✅ 明天改坏了随时可以回来

⏸️ 改了一半，先换个任务？
git stash      # 半成品先存起来
# 去紧急修 bug…
git stash pop  # 恢复回来继续

✅ 半成品完好无损地保存着

↩️ 改错了想反悔

最安全的办法：

git stash      # 当前改动先存起来，可以找回
# 另一种（不能反悔的）：
git checkout -- 文件名  # 放弃改动回到上次commit

✅ 文件回到了干净的状态

🌌 尝试新想法，又怕搞坏
git checkout -b 试试新方案  # 创建+切换到新分支
# 在新分支里随便改…
git add . && git commit -m "在新分支上改的内容"
git checkout main           # 切回主线

觉得好就合并回去：

git merge 试试新方案

✅ 两个平行世界，互不干扰

💻 多台电脑同步

第一次推：

git remote add origin https://github.com/你的用户名/项目名.git
git push -u origin main

另一台电脑首次拿：

git clone https://github.com/你的用户名/项目名.git
# 改完→提交→推
git add . && git commit -m "改了" && git push

✅ 两台电脑自动同步，不用 U 盘或微信传文件

🤖 和 AI Agent 配合
核心原则

main（或 master）分支是"干净的、确定没问题的版本"。 所有实验、所有 AI 干的活，都在别的分支上做。

# AI 干活前，先建分支
git checkout -b ai-实验功能

# 让 AI 在这个分支上随便改

# 觉得好：合并
git checkout main
git merge ai-实验功能

# 觉得不好：删了就行
git branch -D ai-实验功能
🔑 让 AI 帮你做 git 操作也可以：直接对它说"帮我创建新分支叫 feat-XX" "帮我把当前改动 commit""帮我合并分支"。

但记住：AI 改代码之前，一定先切到新分支。这样它搞烂了， 你切回 main，世界立刻恢复原样。

❓ 常见问题
什么是 .gitignore？
有些文件不应该被 Git 存档—— 密码文件、编译生成的文件、系统隐藏文件。 在项目根目录建一个 .gitignore 写上它们。 git add 时自动跳过。
冲突是啥？遇到怎么办？
你改了一行，别人也改了同一行，Git 不知道该听谁的。 打开文件，看到 <<<< >>>> 标记， 决定保留谁的版本，删掉标记，git add . → git commit。不用慌，冲突是正常保护机制。
rebase 是啥？
merge 的另一种形式。merge 是"并列合并"， rebase 是"把你的改动接到别人后面"。新手建议只用 merge， 等用熟了再碰 rebase。
我忘了 git push，换了电脑怎么办？
如果只 commit 没 push，代码只在你原来那台电脑上。 保险做法：干完活就 git push，养成习惯。
