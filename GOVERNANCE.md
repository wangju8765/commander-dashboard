# 主宰面板更新规范

> 所有 Agent 在向主宰面板新增/修改项目时，必须遵守此规范。

---

## 核心流程：完工后三件事

一件事做完了，如果有项目关联，按顺序做以下三件事：

### 第一步：PROJECTS.md（主 workspace 索引）

文件位置：`/Users/zeno/Documents/Openclaw-Workspace/PROJECTS.md`

- **新项目** → 加一行（名称 + 状态 + 负责人 + 目录）
- **状态变更** → 换 emoji（🎯规划中 → 🚧进行中 → ✅已完成 → 📦已归档）
- **一次性交付** → 用 `📄 已交付`，不是 `✅ 已完成`

### 第二步：项目本地文档

- **持续开发项目** → 更新 `LOG.md`（做了什么）+ `PLAN.md`（进度变化）
- **一次性调研项目** → 只写 `README.md` 做交付物说明，不需要 PLAN 和 LOG
- **新项目建目录** → `projects/项目名/` + 必须的文档

### 第三步：主宰面板数据库（Supabase projects 表）

通过 Supabase REST API 更新。

#### 字段规范：

| 字段 | 要求 |
|------|------|
| `name` | 简洁的项目名称 |
| `status` | planning / in_progress / completed / archived |
| `progress` | 0-100 整数 |
| `description` | **三段式 Markdown：** `**背景：**` `**目标：**` `**进展：**` |
| `lead` | 负责人名，必须跟 PROJECTS.md 一致 |
| `tags` | `["标签1","标签2"]` |
| `docs` | 项目产出列表（见下方规范） |

#### docs 字段规范：

```json
[
  {
    "type": "link",
    "title": "文档标题",
    "url": "https://wangju8765.github.io/commander-dashboard/xxx.html"
  }
]
```

- **必须用 `type:"link"`**，指向部署到 GitHub Pages 的独立 HTML 页面
- **不要用 `type:"doc"` 内嵌 Markdown** —— 所有产出统一为独立排版页面
- 如果产出文件尚未部署，先部署再填 URL，不要把空 URL 或未完成的 URL 提交

## 调研报告 → HTML 页面规范

每次将调研报告/教程文档从 Markdown 转为 HTML 时，必须遵循以下要求：

### 前置要求

1. **先分析内容结构**，不是直接套用现有模板
2. **根据内容设计信息层级**：
   - 什么信息应该用对比框/网格/卡片/时间线/流程图/问答列表
   - 不同的内容结构需要不同的视觉方案
3. 可以复用已有的 CSS 变量和基础样式，但**页面布局必须量体裁衣**

### 规范清单

```
□ 内容分析：这份文档的结构是什么？（对比？步骤？问答？故事？）
□ 信息层级设计：主要内容用什么组件呈现？
□ 视觉方案：适配内容结构的布局，不套模板
□ 配色可读：深色背景+浅色文字，高亮色有含义
□ 手机适配：375px 宽度正常阅读，文字不溢出
□ 响应式：必要时支持平板/桌面横屏
□ 页面底部有生成时间和出品标识
```

### 禁止行为

- ❌ 把 Markdown 逐段塞进别人的模板
- ❌ 长代码块展示流程或步骤
- ❌ 信息层次扁平（全部用段落文字平铺）
